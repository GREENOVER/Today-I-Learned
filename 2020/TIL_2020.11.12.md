### 11월 12일 (목)

- #### 학습내용
오늘은 타입과 개체의 속성/기능들을 토대로 개체를 일반화 하는것과 추상화 하는것에 대해 차이를 학습하며 실제로 간단하게 스위프트 언어로도 표현해보며 구성을 해보았다. 야곰이 사전에 제공한 학습자료를 보았을때도 일반화와 추상화에 대해 이해가 되지 않았었다. 실제로 팀원과 실습을 해보며 설명을 들으니 확실한 차이를 느낄 수 있었다. 우선 사전자료를 토대로 개념들을 익히며 정리해본것은 아래와 같다.
  - 클래스 : 객체가 가진 특성/기능을 정리한 틀
  - 속성(변수): 프로퍼티 / 기능(함수): function(메서드)
  - 타입 : 개체를 대표하는 표현, 타입을 실체화 한것이 인스턴스로 메모리에 실제로 할당되는것이다. 이것을 인스턴스화 한다고 말한다. 반대로 생각해보면 타입은 공통된 인스턴스들의 일반화라고 볼 수 있다. 
  - 타입의 종류 : 구조체, 클래스, 열거형, 프로토콜
  - 메서드: 인스턴스 or 타입의 공통된 기능
  - 클로저(코드블럭) > 함수(이름을 가진 클로저) > 메서드(타입에 연관되어 사용하는 함수)
  - 변수: 특정 데이터 해당하는 값의 이름 (임시기억장치 메모리에 저장)
  - 상수: 변수와 같은 의미지만 변경될 수 없는 값 (가독성이 좋아야할때 사용)
  - 프로퍼티: 특정 타입에 연관된 값 (타입에 속한 변수,상수)
  - 일반화: 개체들의 공통된 특성과 기능을 프로퍼티와 메서드로 표현할 수 있다.
  - 추상화: 일반화된 것중에서도 사용자에게 보여질 필요한 관점만 추출하는것으로 대부분의 기능들은 추상화된 기능, 즉 밖으로 드러나야 하는것이다.

- #### 문제점 / 고민한 점
웹 브라우저 스텝 4를 하다 막힌 부분이 있다. 스텝4에서 https가 붙지 않았을때, https를 URL에 붙여줘서 유효한 사이트면 https를 붙여서 접근하게 변경해주는 작업을 진행중이다. 우선 스텝 3의 과정인 사이트의 유효/무효검증하는 함수에서 http일때 https가 가능한지 점검하고 변환해주는 함수를 만들었다. String 내장함수인 replacingOccurrences를 사용하여 http 사이트를 https로 붙여 변환을 해주었으며, 그다음으로 https가 붙은 그 사이트가 실제로 존재하는 사이트인지 확인해주는 기능을 구현하려다 막혔다. 공식 개발문서를 먼저 찾아보았는데 canOpenURL 함수와 handlesURLScheme 함수가 적절한것 같다고 판단하여, 먼저 canOpenURL 함수를 이용하여 시도를 해보았다. 해당 사이트 문자열 주소를 URL로 다시 바꿔준 후 canOpenURL을 통해 bool타입으로 받았다. 정상작동하는듯 하였으나, 잘 알고보니 해당 함수는 존재하는 사이트인지 검증해주는것이 아닌 열릴 수 있는건지 검증해주는 기능이였다. 예를들어 [http://green.green을](http://green.xn--green-ei3w) 값으로 주어도 URL을 통해 열리긴 하니까 보여주는것으로 구현하고자 하는 의도와 달랐다. 여기서 필요한 기능은 404 웹 페이지 에러 같이 없는 사이트를 검증해주는 부분이 필요했다. 그래서 두번째로 handlesURLScheme 함수를 적용해보려 노력했다. 개발문서에 너무 정보가 없어 구글링을 통해 한참을 찾으며 조금 있는 정보로 적용을 시켜봤는데 오류 투성이여서 이렇게 쓰는것이 아니라고 생각했다. 

- #### 해결방법
현재도 풀리지 않은 해당 문제에 대해서 계속 고민중이며 다른 방법으로 시도하려고 노력중이다. 정규식을 사용하는것도 생각했는데 그것도 canOpenURL 처럼 구현하고자하는 검증이 아닐것 같다고 생각이 들었다. 아니면 검증을 하기위해 뷰를 하나 더 생성하고 그 뷰가 열리면 반환 값을 통해 판별해주는 방법도 생각하며 적용시켜보려고 노력하고있다.
